### What it actually shows (nuanced — not a clean "graph wins")

1. **First real MCP usage in the whole POC.** A hard, un-greppable-at-scale task + an explicit steer finally broke the adoption ceiling: cgc/serena/both all called their tools (`mcp>0`). Adoption needs *both* a task where grep obviously fails *and* steering — neither alone did it across R1–A3.
2. **The graph isn't *required* for completeness.** baseline traced the call tree manually to **28/29** — grep-only can do impact analysis, just laboriously (64 turns, $5.90) and slightly incompletely.
3. **Used well, the graph is ~4× cheaper at equal completeness.** `both` hit the full 29/29 in **5 graph calls / 15 turns / $1.51** vs baseline's $5.90 — because it expressed the transitive reachability as one `CALLS*` query instead of dozens of manual steps.
4. **But "graph available" ≠ "graph efficient."** `cgc`-alone had the same graph, used it, yet cost as much as baseline ($6.30) — it didn't lean on the single-query traversal the way `both` did. The decisive variable is *how* the agent uses the graph, not merely having it.
5. **LSP is the wrong shape for transitivity.** serena got there but needed 61 recursive `find_referencing_symbols` calls — the graph's `CALLS*` path query is the natural fit; an LSP has no transitive-closure primitive.

### Caveats
Single run per arm; A4 cell costs ranged $1.5–6.3 and run-to-run variance is large (see R1–R3), so the cost ordering (esp. cgc vs both) is **not** statistically settled — a median over ≥3 runs is needed before quoting "4× cheaper" as firm. The impact set is also test-file-heavy (the summarizer is well-tested). See [`FINDINGS.md`](../FINDINGS.md).
